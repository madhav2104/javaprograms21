SECTION 1 — PRODUCT REQUIREMENTS DOCUMENT (PRD)


Title
[Data Platform][CDM Optimization] - Redesign Customer & System Metadata Collection Model

Objective
The objective of this initiative is to standardize and optimize the collection, storage, and management of customer and system metadata across all enterprise data pipelines. By eliminating redundant data storage and enforcing a unified schema, the project aims to reduce operational costs, improve data quality, and enhance governance. The solution will focus on creating canonical entities for customer and system metadata, ensuring that personally identifiable information (PII) is handled in compliance with governance policies, and enabling seamless data integration and analytics across CRM, ERP, and internal services.

Background
Currently, customer and system metadata are independently collected and stored by multiple systems, including CRM, ERP, and various internal services. This decentralized approach has led to significant data duplication, with attributes such as customerId, customerName, email, systemId, systemVersion, and serviceName being repeatedly embedded in every event and transaction, regardless of whether their values have changed. The lack of a global identifier for customers (with CRM using Customer_ID and ERP using CustID) further complicates data integration and analysis. This fragmentation not only increases storage and processing costs but also results in inconsistent schema definitions, making it difficult to maintain data governance and to provide a unified view of the customer for business stakeholders such as the marketing team.

Problem Statement
The current model of embedding full customer and system metadata in every event has resulted in excessive data duplication, increased storage and data transfer costs, and inconsistent schema definitions across systems. The absence of a canonical global identifier for customers hinders the ability to join datasets effectively, while the repeated inclusion of PII fields such as email and phone in transactional tables raises compliance and governance concerns. These issues collectively impede the organization's ability to maintain a unified, high-quality data platform, complicate data governance, and limit the effectiveness of business analytics and segmentation efforts.

Functional Requirements
The redesigned data model must establish a centralized Customer Master entity and a System Metadata entity, each serving as the authoritative source for their respective domains. All transactional and event data should reference these entities via foreign keys rather than duplicating metadata attributes. The model must enforce a single, canonical customer_id across all systems, resolving existing naming conflicts (e.g., Customer_ID in CRM, CustID in ERP) into a unified identifier. PII fields such as email and phone must be stored exclusively in the Customer Master and masked according to governance rules; these fields must not appear in transactional tables. System metadata attributes (systemId, systemVersion, serviceName) should be similarly centralized. The solution must support schema evolution and allow for updates to master entities only when changes occur or at defined intervals. All data pipelines and services must adopt the standardized schema, and integration points must be updated to reference the master entities. The model should facilitate efficient joins and enable a unified customer view for business analytics.

Non-Functional Requirements
The solution must ensure scalability to handle high event volumes and support future growth in both data volume and complexity. Data governance and compliance, particularly regarding PII masking and access controls, must be strictly enforced. The architecture should be compatible with existing data platforms, such as Databricks, and support efficient data ingestion and processing. Schema consistency and versioning must be maintained to ensure backward compatibility and facilitate smooth integration across systems. The model should minimize storage and processing overhead and be resilient to changes in upstream or downstream systems.

Proposed Solution
The recommended approach is to implement a centralized Canonical Data Model (CDM) comprising a Customer Master entity and a System Metadata entity. Transactional and event data will reference these entities via foreign keys, eliminating the need to duplicate metadata attributes in every event. The Customer Master will resolve naming conflicts by enforcing a single customer_id, and will store all PII fields with appropriate masking and governance controls. The System Metadata entity will centralize system-related attributes, ensuring consistency and reducing redundancy. Data pipelines will be refactored to adopt the new schema, and integration points will be updated to reference the master entities. The model will support schema evolution and versioning, and will be designed for compatibility with Databricks and other enterprise data platforms. Governance policies will be enforced to ensure that PII is not present in transactional tables and that all data access is properly controlled and audited.

Assumptions
It is assumed that all relevant systems (CRM, ERP, internal services) can be updated to adopt the new canonical schema and that integration points can be modified to reference the master entities. It is also assumed that a single, canonical customer_id can be established and propagated across all systems, and that governance policies regarding PII masking and access can be enforced at the platform level. Where information is missing, such as the specific technical mechanisms for schema enforcement or the precise update intervals for master entities, these are considered open points to be addressed during detailed design and implementation.
